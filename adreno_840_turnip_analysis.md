# Xenia on Adreno 840 (Snapdragon 8 Elite/Gen 4) with Mesa Turnip Driver: Architectural Friction & Proposed Optimizations

This document outlines an architectural analysis of running the Xenia Xbox 360 emulator's Vulkan backend on the Adreno 840 GPU, focusing on the specific friction points between the Xbox 360's hardware design and modern mobile Tile-Based Deferred Rendering (TBDR) architectures. It identifies the root causes of graphical glitches (such as texture corruption, flickering, and low performance) often observed on this hardware, and proposes specific C++ and Vulkan code changes for optimization.

---

## 1. Architectural Friction Points

The core issue stems from emulating the Xbox 360's unique graphics pipeline on a modern mobile GPU.

### A. The Xbox 360 Architecture (Xenos GPU)
The Xbox 360 GPU is fundamentally an Immediate-Mode (IM) renderer, but it features a very specific memory architecture: a 10 MB block of embedded DRAM (eDRAM) connected directly to the GPU die.
*   **Massive Bandwidth:** This eDRAM allowed for incredibly fast reads and writes for color, depth, and stencil operations.
*   **"Free" Multisample Anti-Aliasing (MSAA):** Developers heavily utilized this fast memory to perform complex blending, alpha testing, and MSAA with almost zero performance penalty.
*   **Tile-Based Rendering (But Not TBDR):** Because 10 MB wasn't enough to hold a full 720p frame buffer with MSAA, games often rendered the screen in "tiles" (e.g., top half, then bottom half), resolving each tile from the eDRAM to main system memory.

### B. The Adreno 840 Architecture (TBDR)
The Qualcomm Adreno 840 (and mobile GPUs in general) uses a Tile-Based Deferred Rendering (TBDR) architecture designed for extreme power efficiency.
*   **On-Chip GMEM:** The screen is divided into small tiles. For each tile, all geometry is binned, and then the tile is rendered entirely within a small, extremely fast on-chip memory (GMEM).
*   **Costly Flushes:** Once a tile is fully processed (all opaque and transparent geometry drawn, depth resolved), the final result is written out to the much slower main system memory (RAM).
*   **The Golden Rule:** The cardinal rule of TBDR optimization is to *keep data in GMEM as long as possible* and minimize flushes (resolves) to system memory.

### C. The Conflict in Xenia's Vulkan Backend
Xenia was originally designed with desktop GPUs (NVIDIA/AMD) in mind, which are pure Immediate-Mode renderers with massive memory bandwidth.
*   **eDRAM Emulation:** Xenia emulates the Xbox 360's eDRAM using a large Storage Buffer (SSBO) or Image in Vulkan (`vulkan_render_target_cache.cc`).
*   **Fragment Shader Interlock:** To accurately emulate the Xbox 360's pipeline, Xenia often relies on `VK_EXT_fragment_shader_interlock`. This extension serializes access to the emulated eDRAM buffer when overlapping fragments are rendered, ensuring correct blending and depth-testing order.
*   **The Problem:** On the Adreno 840, forcing serialization (Interlock) or issuing frequent explicit memory barriers to synchronize the emulated eDRAM buffer completely breaks the TBDR pipeline. It forces the GPU to constantly flush its fast GMEM to system RAM, stall, and then reload it. This "GMEM thrashing" causes severe performance drops. Furthermore, the official Qualcomm drivers often contain bugs when handling complex compute shaders or fragment interlocks, leading to the "rainbow textures" or flickering glitches.

## 2. Identified Bottlenecks and Missing Workarounds

### A. Subpass Inefficiencies
Xenia frequently starts and stops Vulkan Render Passes (`vkCmdBeginRenderPass` / `vkCmdEndRenderPass`) or injects pipeline barriers when the emulated Xbox 360 game changes its render state or flushes its eDRAM.
*   **Adreno Impact:** Every `vkCmdEndRenderPass` forces a flush of GMEM to system memory.
*   **Solution:** Vulkan provides `VkSubpassDescription` and `VkSubpassDependency` to allow multiple dependent rendering steps to occur *within* the same Render Pass, keeping the data in GMEM.

### B. Missing `VK_EXT_shader_tile_image` Utilization
When an Xbox 360 game reads back from the eDRAM (e.g., for depth-testing in a subsequent pass), Xenia currently reads from the global emulated buffer.
*   **Adreno Impact:** This forces a GMEM flush and a slow read from system RAM.
*   **Solution:** The Adreno 840 (especially with the Mesa Turnip driver) supports extensions like `VK_EXT_shader_tile_image` (or Qualcomm's proprietary `VK_QCOM_tile_shading`). This allows a fragment shader to read the color/depth data directly from the current tile in GMEM, drastically reducing memory bandwidth and bypassing the need to flush.

### C. The Need for the Mesa Turnip Driver
The stock Qualcomm drivers prioritize common mobile games (like Genshin Impact) and often fail on the complex, low-level Vulkan instructions generated by heavy emulators like Xenia.
*   **Turnip:** The open-source Mesa Turnip driver is specifically designed to correctly implement these complex Vulkan specifications, fixing graphical corruption and preventing stutters. Xenia should detect Turnip and optimize its paths accordingly.

## 3. Proposed Code Changes (For Implementation and Testing)

To optimize Xenia for the Adreno 840, specific code adjustments must be made to the Vulkan backend to reduce GMEM thrashing and improve subpass utilization.

### Code Adjustment 1: Turnip/Adreno Detection

We need to identify when we are running on an Adreno GPU and specifically the Turnip driver, so we can enable our TBDR-specific optimizations.

**File:** `app/src/main/cpp/xenia/src/xenia/gpu/vulkan/vulkan_graphics_system.cc` (or a relevant initialization file)

```cpp
// Within VulkanGraphicsSystem::Setup or similar device initialization:

bool is_turnip_driver = false;
bool is_adreno_gpu = false;

VkPhysicalDeviceProperties properties;
vkGetPhysicalDeviceProperties(physical_device_, &properties);

// Detect Adreno hardware
if (properties.vendorID == 0x5143) { // Qualcomm Vendor ID
    is_adreno_gpu = true;
    XELOGI("Detected Qualcomm Adreno GPU.");
}

// Check for the Turnip driver name in properties.deviceName or driver properties if available via VK_KHR_driver_properties
if (strstr(properties.deviceName, "Turnip") != nullptr || strstr(properties.deviceName, "Mesa") != nullptr) {
    is_turnip_driver = true;
    XELOGI("Detected Mesa Turnip driver. Enabling TBDR optimizations.");
}

// Store these flags in the VulkanCommandProcessor or a global configuration object
command_processor_->SetTurnipOptimizationsEnabled(is_adreno_gpu && is_turnip_driver);
```

### Code Adjustment 2: Optimizing Render Pass Dependencies (Avoiding GMEM Flushes)

Currently, `vulkan_render_target_cache.cc` creates external subpass dependencies that might be overly conservative for TBDR. We can tweak the dependencies to hint to the Adreno driver that data should remain in GMEM.

**File:** `app/src/main/cpp/xenia/src/xenia/gpu/vulkan/vulkan_render_target_cache.cc`

Look for the creation of `VkSubpassDependency` (e.g., around the fragment shader interlock setup or general render pass creation).

```cpp
// Modifying the subpass dependencies to be more TBDR-friendly.
// We want to avoid VK_PIPELINE_STAGE_BOTTOM_OF_PIPE_BIT or heavy memory barriers
// between subpasses if they can be resolved By-Region (within the tile).

VkSubpassDependency dependencies[2];

// Dependency 0: External to Subpass 0
dependencies[0].srcSubpass = VK_SUBPASS_EXTERNAL;
dependencies[0].dstSubpass = 0;
// Wait for earlier fragment shading to finish before we start fragment shading in this pass.
dependencies[0].srcStageMask = VK_PIPELINE_STAGE_FRAGMENT_SHADER_BIT | VK_PIPELINE_STAGE_COLOR_ATTACHMENT_OUTPUT_BIT;
dependencies[0].dstStageMask = VK_PIPELINE_STAGE_FRAGMENT_SHADER_BIT | VK_PIPELINE_STAGE_COLOR_ATTACHMENT_OUTPUT_BIT;
dependencies[0].srcAccessMask = VK_ACCESS_SHADER_WRITE_BIT | VK_ACCESS_COLOR_ATTACHMENT_WRITE_BIT;
dependencies[0].dstAccessMask = VK_ACCESS_SHADER_READ_BIT | VK_ACCESS_COLOR_ATTACHMENT_READ_BIT;
// CRITICAL FOR TBDR: VK_DEPENDENCY_BY_REGION_BIT tells the driver that the dependency
// is only localized to the pixel coordinate. This allows the Adreno GPU to keep
// the tile in GMEM rather than flushing the entire framebuffer to sync.
dependencies[0].dependencyFlags = VK_DEPENDENCY_BY_REGION_BIT;

// Dependency 1: Subpass 0 to External
dependencies[1].srcSubpass = 0;
dependencies[1].dstSubpass = VK_SUBPASS_EXTERNAL;
dependencies[1].srcStageMask = VK_PIPELINE_STAGE_COLOR_ATTACHMENT_OUTPUT_BIT;
dependencies[1].dstStageMask = VK_PIPELINE_STAGE_BOTTOM_OF_PIPE_BIT;
dependencies[1].srcAccessMask = VK_ACCESS_COLOR_ATTACHMENT_WRITE_BIT | VK_ACCESS_COLOR_ATTACHMENT_READ_BIT;
dependencies[1].dstAccessMask = VK_ACCESS_MEMORY_READ_BIT;
// Again, localized dependency to prevent full pipeline stalls if possible.
dependencies[1].dependencyFlags = VK_DEPENDENCY_BY_REGION_BIT;
```

// Check if the physical device supports reading from the tile image directly.
